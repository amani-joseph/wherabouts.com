## Task 2 — `searchPlaces` ranked query

**Files:** `packages/database/src/queries/search-places.ts` (create),
`packages/database/src/queries/index.ts` (edit — export it)

1. Implement `searchPlaces(db, query, opts)` where `opts` = `{ category?, latitude?, longitude?,
   limit }`, bound to the `places` table via the `tiered-search.ts` primitives.
2. Apply the `category` filter inside `buildWhereClause` (equality on `category`), alongside the
   fuzzy match on `search_text`.
3. Proximity: when `latitude`/`longitude` are passed, order by `geom <-> point` (KNN) after the
   fuzzy match narrows candidates — identical pattern to `buildOrderBy`. Otherwise order by
   `similarity_score DESC, population_score DESC`.
4. Return a typed `PlaceSearchResult[]` (`name`, `category`, `locality`, `latitude`,
   `longitude`, `similarity_score`, `osmType`/`osmId`).

**Verify:** unit test against the seeded sample from Plan 01: a text query returns the expected
POI ranked first; adding `category` excludes off-category rows; adding lat/lng near a POI floats
it up the ranking.

**Commit:** `feat(db): tiered POI search query (searchPlaces) reusing tier engine`

## Task 3 — `/api/v1/places/search` oRPC endpoint + tests

**Files:** `packages/api/src/routers/public/places-query.ts` (create),
`packages/api/src/routers/public/places.ts` (create),
`packages/api/src/routers/public/index.ts` (edit — register),
`packages/api/src/routers/public/places.test.ts` (create)

1. `places-query.ts`: pure, env-free helper(s) (e.g. `buildPlacesQuery` for normalising the
   text query / trimming) mirroring `geocode-query.ts`, so they unit-test without `serverEnv`.
2. `places.ts`: oRPC procedure modelled on `forwardGeocode`:
   - zod input: `q` (`z.string().min(1)`), `category` (`z.enum(PLACE_CATEGORIES).optional()`),
     `lat`/`lng` (`z.coerce.number().optional()`), `limit` (`z.coerce.number().int().min(1)
     .max(50).default(10)`). **Use `z.coerce`** for all numerics (GET coercion gotcha).
   - `.use(apiKeyAuth)`, `.use(usageMiddleware("places.search"))`.
   - OpenAPI `route`: `method GET`, `path "/api/v1/places/search"`, `summary`, `tags: ["places"]`.
   - Delegate to `searchPlaces`; map rows to the public output shape; throw `ORPCError` NOT_FOUND
     semantics consistent with geocode when empty (or return empty list — match geocode's
     convention; document choice).
3. Register the procedure in `routers/public/index.ts` next to the addresses/geocode procedures.
4. Tests (`places.test.ts`) mirror `geocode.test.ts`: assert input validation (bad lat 400 path,
   coercion works), category filter passed through, and the output envelope shape.

**Verify (criterion 2):**
- `GET /api/v1/places/search?q=cafe&lat=-33.86&lng=151.2&limit=5` returns ranked results.
- `&category=retail` restricts to retail POIs.
- Endpoint appears in the generated OpenAPI doc under `places` tag.
- `pnpm -F @wherabouts.com/api test` green.
