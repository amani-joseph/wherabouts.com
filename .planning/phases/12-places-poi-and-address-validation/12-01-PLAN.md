## Task 2 — `places` schema + migration

**Files:** `packages/database/src/schema/places.ts` (create),
`packages/database/src/schema/index.ts` (edit), `packages/database/drizzle/0013_places.sql`
(generated)

1. Create `places` table mirroring `addresses.ts` patterns:
   - `id` identity PK; `osmType` (varchar 1: n/w/r), `osmId` (bigint) — **UNIQUE together**
     (natural key for idempotent upsert).
   - `name text notNull`, `category varchar` (one of `PLACE_CATEGORIES`), `tags jsonb`
     (raw OSM tags for later refinement).
   - `country` (2), `state` (10), `locality`, `postcode` (10) — nullable (OSM coverage varies).
   - `longitude`/`latitude` real notNull, `geom geometry(Point,4326)` (reuse the `customType`
     geometry from `addresses.ts` — extract it to a shared module if cleaner).
   - `searchText text` (name + category + locality, normalised), `populationScore integer
     default 0`.
   - Indexes: `unique(osmType, osmId)`; btree on `category`, `(country,state)`,
     GiST on `geom`, GiST on `(geom::geography)` (for `ST_DWithin`), and a **trigram** index on
     `search_text` (GIN/GiST `gin_trgm_ops`) matching the addresses tiered-search setup.
2. Export `Place` / `NewPlace` types; add to the schema barrel `index.ts`.
3. Generate the migration: `pnpm -F @wherabouts.com/database drizzle-kit generate` → produces
   `drizzle/0013_places.sql`. **Do NOT hand-write SQL out-of-journal** (project memory). Confirm
   the generated SQL `CREATE EXTENSION IF NOT EXISTS pg_trgm` is present or already installed by
   `0009_tiered_search_extensions.sql`; if the trigram index needs `gin_trgm_ops`, verify the
   generated DDL includes it (drizzle may need a manual `.using("gin", sql\`...gin_trgm_ops\`)`).

**Verify:** `drizzle-kit generate` produces `0013_places.sql` with `CREATE TABLE`; `pnpm -F
@wherabouts.com/database check`/typecheck passes; schema barrel exports `places`.

**Commit:** `feat(db): add places (POI) table with PostGIS + trigram indexes`

</task>

## Task 3 — Idempotent OSM POI importer

**Files:** `scripts/import-osm-pois.ts` (create),
`packages/database/scripts/README-osm-pois.md` (create — operator runbook)

1. Model the script on `scripts/import-gnaf.ts`: read an AU OSM source (Geofabrik
   `australia-latest.osm.pbf` parsed to GeoJSON/NDJSON via osmium, OR a pre-extracted NDJSON of
   POIs — document the chosen pipeline in the README). Stream, do not load all in memory.
2. For each named POI node/way-centroid: derive `category` via `mapOsmTagsToCategory`, build
   `search_text`, compute `geom` via `ST_SetSRID(ST_MakePoint(lng,lat),4326)`.
3. **Batch-idempotent write** (neon-http has NO transactions): chunk rows (~1-5k) and
   `INSERT ... ON CONFLICT (osm_type, osm_id) DO UPDATE SET ...`. No `db.transaction()`, no
   `FOR UPDATE`. Re-running must be a no-op delta, not a duplicate.
4. Log progress counts; make chunk size + input path CLI args.
5. README documents: download URL, osmium extraction command, env (`DATABASE_URL`), how to run,
   expected row count, ODbL attribution requirement.

**Verify (criterion 1):**
- Run the importer over a **small AU sample extract** (e.g. one suburb bbox) end-to-end.
- `SELECT count(*) FROM places` > 0; `SELECT category, count(*) FROM places GROUP BY 1` shows a
  sane distribution.
- **Re-run the same extract → row count unchanged** (idempotency proof).
- `SELECT * FROM places ORDER BY geom <-> ST_SetSRID(ST_MakePoint(lng,lat),4326) LIMIT 5` returns
  rows (proves geom + GiST usable by Plan 02 proximity).

**Commit:** `feat(scripts): idempotent AU OSM POI importer into places`
