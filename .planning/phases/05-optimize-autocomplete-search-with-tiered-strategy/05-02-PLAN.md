<objective>
Rewrite the autocomplete query function to implement tiered search strategy based on input length, with result ranking by population, admin level, and similarity score.

Purpose: The current query uses a single ILIKE pattern match which is slow on 60M rows and returns unranked results. The tiered approach uses progressively more expensive search techniques as input length increases, keeping short queries fast while enabling fuzzy matching for longer inputs.

Output: A rewritten `autocompleteAddresses` function that dispatches to the appropriate search tier.

**Tiered dispatch logic** (based on trimmed query length):

```typescript
const len = trimmed.length;

if (len < 3) {
  // Tier 0: Too short — return empty
  return [];
}
```

For len >= 3, build the query using raw SQL via Drizzle's `sql` template literal (import `sql` from `drizzle-orm`). The strategy:

**Tier 1 (3-4 chars): Prefix search only**
```sql
SELECT *, 1.0 as similarity_score
FROM addresses
WHERE search_text LIKE ${trimmed + '%'}
  AND (country filter if provided)
  AND (state filter if provided)
ORDER BY population_score DESC, admin_level ASC
LIMIT ${limit}
```
This uses the B-tree `text_pattern_ops` index for fast prefix matching.

**Tier 2 (5-7 chars): Trigram similarity + fuzzy fallback**
```sql
SELECT *,
  similarity(search_text, ${trimmed}) as similarity_score
FROM addresses
WHERE search_text % ${trimmed}
  AND (country filter if provided)
  AND (state filter if provided)
ORDER BY
  population_score DESC,
  admin_level ASC,
  similarity_score DESC
LIMIT ${limit}
```
The `%` operator uses the GIN trigram index. `similarity()` threshold defaults to 0.3.

If zero results, fall back to levenshtein:
```sql
SELECT *, 0.5 as similarity_score
FROM addresses
WHERE levenshtein(lower(left(search_text, ${len + 2})), lower(${trimmed})) <= 1
  AND (country filter if provided)
LIMIT ${limit}
```
Note: `left(search_text, len+2)` truncates search_text to near-input length so levenshtein is efficient (it operates on short strings).

**Tier 3 (8+ chars): Trigram with wider fuzzy tolerance**
Same as Tier 2 but:
- Use `word_similarity(search_text, ${trimmed})` instead of `similarity()` for partial word matching
- Use `<%%` operator (word similarity) which is better for longer strings matching against long search_text
- Levenshtein fallback uses distance <= 2 instead of <= 1
- If levenshtein also returns zero, apply phonetic fallback:
```sql
SELECT *, 0.3 as similarity_score
FROM addresses
WHERE dmetaphone(left(search_text, 20)) = dmetaphone(${trimmed})
  AND (country filter if provided)
LIMIT ${limit}
```

**Proximity boosting** (when latitude/longitude provided):
Add to ORDER BY after population_score:
```sql
ST_Distance(
  geom,
  ST_SetSRID(ST_MakePoint(${longitude}, ${latitude}), 4326)
) ASC
```
Only add this clause if both lat and lon are provided.

**Implementation approach:**
- Use Drizzle's `sql` tagged template for raw queries since trigram operators and functions cannot be expressed in Drizzle's query builder
- Use `db.execute(sql`...`)` which returns raw rows
- Map raw rows through `formatStreetAddress` and construct `AutocompleteResult` objects
- Keep country/state filters as parameterized values (never interpolate user input into SQL)
- Set `pg_trgm.similarity_threshold` to 0.3 at start of query: `SET pg_trgm.similarity_threshold = 0.3;` or use `set_limit(0.3)` before the SELECT
- Add type for raw row result to map columns properly
