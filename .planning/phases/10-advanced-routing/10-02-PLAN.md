<objective>
Add reachability isochrones: sample points around an origin, ask OSRM `/table` how long each
takes, keep the ones inside the budget, and shrink-wrap them into a polygon with PostGIS —
optionally reporting which ABS regions the polygon overlaps.

Purpose: Isochrones are the marquee parity feature competitors charge for and the only deliverable
that combines OSRM + PostGIS. **Resolve D3 (ConcaveHull vs ConvexHull on Neon) and D4 (sampling
density) before finalising the query.**

Output: `/api/v1/routing/isochrone` returning a GeoJSON reachability polygon, with an optional
ABS-region overlap, plus unit tests.
</objective>

## Task 1 — Probe Neon PostGIS hull capability (D3)

**Human/agent checkpoint — decide the hull function before writing the query.**

1. Query the live Neon DB once: `SELECT postgis_full_version();` and read the GEOS version.
   - GEOS ≥ 3.11 ⇒ use `ST_ConcaveHull(ST_Collect(pts), 0.3)` (concaveness tunable 0.3→0.1).
   - Otherwise ⇒ fall back to `ST_ConvexHull(ST_Collect(pts))` and document the over-estimation
     caveat. (Note: neon-http has no transactions — this is a single read, fine.)
2. Record the result and the chosen function in this plan's SUMMARY **and** update CONTEXT.md D3
   with the resolved decision.

**Verify:** the decision is written down; the chosen SQL function name is fixed for Task 3.

**Commit:** none (investigation) — capture in SUMMARY.

## Task 2 — Sample-point generation (D4)

**Files:** `packages/api/src/shared/isochrone-queries.ts` (create),
`packages/api/src/shared/isochrone-queries.test.ts` (create)

1. `generateSamplePoints(origin: LatLng, opts)` — produce a deterministic ring/grid of
   destination points around the origin: `N` bearings × `M` radius steps (radii scaled to the
   budget, e.g. a max-reach estimate). Total points **must stay within the `/table` cap** from
   plan 01 (document the constant, e.g. ≤ ~100 coords incl. origin). Pure function — no I/O.
2. Use a top-level constant for bearings/steps (no magic numbers; lint rule). Compute offsets with
   a simple equirectangular approximation (metres → degrees) — adequate for AU latitudes.
3. Tests: point count = `N*M`, all within the cap; points are distinct; symmetry sanity (a ring
   at radius r has the expected count).

## Task 3 — Reachable-set filter + PostGIS hull + ABS overlap

**Files:** `packages/api/src/shared/isochrone-queries.ts` (edit),
`packages/api/src/shared/isochrone-queries.test.ts` (edit)

1. `reachablePoints(durations, samplePoints, budgetSeconds)` — keep sample points whose
   origin→point duration (from the OSRM `/table` first row) is `<= budgetSeconds` and not `null`.
   Always include the origin. Pure function (durations passed in) → unit-testable without OSRM.
2. `hullPolygon(db, points)` — build a PostGIS geometry from the reachable points and return its
   hull as GeoJSON, using the function chosen in Task 1:
   `ST_AsGeoJSON(<HULL>(ST_Collect(ST_SetSRID(ST_MakePoint(lng,lat),4326))))`. Follow the raw
   `sql\`...\`` pattern from `region-queries.ts`. Guard the degenerate case (< 3 reachable points
   ⇒ return a small buffered point or a clear `no_isochrone` error rather than an invalid polygon).
3. `regionsOverlappingIsochrone(db, polygonGeoJson, layers?)` (optional overlap) — reuse the
   `regions` table: `ST_Intersects(regions.geom, ST_GeomFromGeoJSON(...))`, grouped via the
   existing `groupRegionsByLayer`. Mirror `regionsContainingPoint`'s structure.
4. Tests: `reachablePoints` filters correctly incl. `null` cells and the budget boundary; hull/
   overlap helpers can be covered with a thin DB integration test or a documented manual check if
   the suite has no live DB (state which in the SUMMARY).

**Verify:** `pnpm -F @wherabouts.com/api test isochrone-queries` green.

**Commit:** `feat(api): add reachable-set filter, PostGIS isochrone hull, and ABS overlap`

</task>

## Task 4 — `/api/v1/routing/isochrone` endpoint (SC #3)

**Files:** `packages/api/src/routers/public/routing.ts` (edit),
`packages/api/src/routers/public/routing.test.ts` (edit),
`packages/api/src/routers/public-http.ts` (edit)

1. Add `routingIsochrone`: `apiKeyAuth` + `usageMiddleware("routing.isochrone")`,
   `path: "/api/v1/routing/isochrone"`, `tags: ["routing"]`.
   - Input (per D5; isochrone input is small so **GET is fine** — use `z.coerce` for all numeric
     params): `origin` (`"lat,lng"` or `addressId`), `profile` (`driving|walking|cycling`,
     default driving), exactly one budget: `durationSeconds` (`z.coerce.number()`) **or**
     `distanceMeters`, and optional `includeRegions` (boolean) + `layers`.
   - Flow: resolve origin → `generateSamplePoints` → `fetchOsrmTable(origin + samples,
     { profile, sources:[0] })` → `reachablePoints` → `hullPolygon` → (optional)
     `regionsOverlappingIsochrone`.
   - Return `{ query: { origin, profile, durationSeconds? }, polygon: <GeoJSON Polygon>, regions? }`.
   - Errors: OSRM down → `INTERNAL_SERVER_ERROR`; too-few reachable points → `UNPROCESSABLE_CONTENT`.
2. Register in `public-http.ts` under `routing:` as `isochrone: routingIsochrone` (GET ⇒ may also
   be added to the API-explorer GET allowlist).
3. Tests: happy path (mock `/table` so a known subset is reachable, assert a Polygon geometry is
   returned); `includeRegions=true` returns the overlap object; OSRM failure → 500; degenerate
   reach → 422; invalid (both/neither budget supplied) → 400.

**Verify:** `pnpm -F @wherabouts.com/api test routing` green; `typecheck` clean.

**Commit:** `feat(api): add /api/v1/routing/isochrone reachability polygons`
