## Task 1 — Add request/response types + methods to the routing resource

**Files:** `packages/sdk/src/resources/routing.ts` (edit)

For each new endpoint, add an exported `*Params` and `*Response` interface mirroring the API
contract from plans 01–04, and add the method to `RoutingResource` + `createRouting`. Follow the
existing `directions` method and the `regions.ts` resource as the shape template; use the
`Requester` contract (`method`, `path`, `query` for GET, `body` for POST, `...options`).

1. **`matrix`** — `MatrixParams { profile?; sources; destinations }`,
   `MatrixResponse { query; durations: (number|null)[][]; distances: (number|null)[][]; sources; destinations }`.
   Method/shape must match what plan 01 chose (GET+query or POST+body per D5).
2. **`isochrone`** — `IsochroneParams { origin?|originAddressId?; profile?; durationSeconds?|distanceMeters?; includeRegions?; layers? }`,
   `IsochroneResponse { query; polygon: GeoJsonPolygon; regions? }`. GET + query.
3. **`match`** — `MatchParams { profile?; coordinates: {lat;lng;timestamp?;radius?}[]; gaps?; tidy? }`,
   `MatchResponse { query; matchings: {...; confidence; geometry}[]; tracepoints: (Tracepoint|null)[] }`.
   POST + body.
4. **`optimize`** — `OptimizeParams { profile?; waypoints: ({lat;lng}|{addressId})[]; roundtrip?; source?; destination? }`,
   `OptimizeResponse { query; trips: {distance_m;duration_s;geometry}[]; waypoints: {order; location}[] }`.
   POST + body.

Reuse the existing `DirectionsGeometry` (`LineString`) type; add a `GeoJsonPolygon` type for the
isochrone. Keep the public `profile` union `"driving" | "walking" | "cycling"` and **update the
existing `DirectionsParams.profile`** from `"driving"` to that union (it was narrowed before
plan 01 widened the API).

**Verify:** `pnpm -F @wherabouts/sdk typecheck` clean; `pnpm -F @wherabouts/sdk build` emits dist.

**Commit:** `feat(sdk): add matrix, isochrone, match, and optimize routing methods`
