<objective>
Add GPS map-matching: snap a noisy coordinate trace to the road network via OSRM `/match`,
returning the matched route geometry and per-point tracepoints (with outliers preserved as null).

Purpose: Map-matching is the input quality layer for fleet/telematics customers and is a thin
wrapper over OSRM `/match` once the profile-aware client (plan 01) exists. **Resolve D5 (GET vs
POST) — recommended POST here because GPS traces can be long.**

Output: `/api/v1/routing/match` plus `fetchOsrmMatch` and a new `no_match` error kind, with tests.

## Task 1 — `fetchOsrmMatch` + `no_match` error kind

**Files:** `packages/api/src/shared/routing-queries.ts` (edit),
`packages/api/src/shared/routing-queries.test.ts` (edit)

1. Extend `RoutingErrorKind` to `"no_route" | "no_match" | "unavailable"`. (Existing call sites are
   exhaustive on the kinds they handle, so adding a kind is safe — verify the `directions` handler
   still compiles; it only checks `no_route` then falls through.)
2. Add `fetchOsrmMatch(trace, options & { profile, timestamps?, radiuses?, gaps?, tidy? })`:
   - Build the `;`-joined `{lng},{lat}` coordinate string.
   - Call `/match/v1/{osrmProfile}/{coords}` with `geometries=geojson`, `overview=full`, and the
     optional `timestamps`/`radiuses` (`;`-joined), `gaps` (`split|ignore`), `tidy` (`true|false`).
   - On `code === "Ok"` return `{ matchings: [{ distance_m, duration_s, confidence, geometry }],
     tracepoints: (Tracepoint | null)[] }` — **keep `null` tracepoints as null** (OSRM outliers).
   - On `code === "NoMatch"` throw `RoutingError("no_match", ...)`.
   - On non-200 / other non-Ok throw `RoutingError("unavailable", ...)`.
3. Tests (extend the `osrmFetch` mock): happy path returns matchings + tracepoints incl. a `null`;
   `NoMatch` → `RoutingError("no_match")`; non-200 → `unavailable`; URL contains `/match/v1/car/`
   and forwards `radiuses`/`timestamps` when supplied.

**Verify:** `pnpm -F @wherabouts.com/api test routing-queries` green.

**Commit:** `feat(api): add fetchOsrmMatch and a no_match routing error kind`

## Task 2 — `/api/v1/routing/match` endpoint (SC #4)

**Files:** `packages/api/src/routers/public/routing.ts` (edit),
`packages/api/src/routers/public/routing.test.ts` (edit),
`packages/api/src/routers/public-http.ts` (edit)

1. Add `routingMatch`: `apiKeyAuth` + `usageMiddleware("routing.match")`,
   `path: "/api/v1/routing/match"`, `tags: ["routing"]`. **POST** per D5.
   - Input (JSON body — validates normally, no `z.coerce` needed): `profile`
     (`driving|walking|cycling`, default driving), `coordinates: [{lat,lng,timestamp?,radius?}]`
     with `min(2)`, optional top-level `gaps` (`split|ignore`), `tidy` (boolean). Reject if any
     point is out of lat/lng range.
   - Split the per-point `timestamp`/`radius` into the parallel arrays `fetchOsrmMatch` expects;
     timestamps must be monotonically increasing (OSRM requirement) → validate and 400 otherwise.
   - Call `fetchOsrmMatch`, return `{ query: { profile }, matchings, tracepoints }`.
   - Errors: `no_match` → `UNPROCESSABLE_CONTENT`; `unavailable` → `INTERNAL_SERVER_ERROR`.
2. Register in `public-http.ts` under `routing:` as `match: routingMatch`. POST ⇒ docs-only in the
   GET-only API-explorer allowlist (do not add to the GET allowlist).
3. Tests: happy path (mock returns matchings + a null tracepoint, assert it survives); `< 2`
   coords → 400; non-monotonic timestamps → 400; `NoMatch` → 422; OSRM down → 500.

**Verify:** `pnpm -F @wherabouts.com/api test routing` green; `typecheck` clean.

**Commit:** `feat(api): add /api/v1/routing/match GPS map-matching endpoint`
