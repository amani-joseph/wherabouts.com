<objective>
Make `Idempotency-Key` a real guarantee on writes: a retried mutation with the same key must
**not** execute twice — it replays the original response. The TS SDK already auto-sends a stable
key across retries (`http.ts`, `http-resilience.test.ts`); today the server ignores it, so an
SDK retry of a transient `503` on `POST /api/v1/zones` creates duplicate rows.

Purpose: Makes Success Criterion 3 TRUE — the highest-value correctness fix in the phase.

Constraint (locked, from memory): the `neon-http` driver has **no transactions** —
`db.transaction()` throws. Enforcement therefore uses an idempotent upsert
(`INSERT ... ON CONFLICT DO NOTHING`): the first writer claims the key, concurrent/duplicate
callers lose the insert race and either replay the stored response (completed) or get `409`
(still in-flight). No row locks.

Output: an `idempotency_keys` table + migration, a claim/complete helper module, an
`idempotencyMiddleware` in the public write path, and tests proving single-execution + replay.
</objective>

## Task 2 — claim / complete helpers (no transactions)

**Files:** `packages/api/src/idempotency.ts`, `packages/api/src/idempotency.test.ts`

1. `hashRequest(method, path, body) -> string` (sha-256 hex via WebCrypto — Workers-compatible).
2. `claimIdempotencyKey(db, { apiKeyId, key, requestHash }) -> ClaimResult` where `ClaimResult`
   is one of:
   - `{ outcome: "claimed" }` — our `INSERT ... ON CONFLICT DO NOTHING` inserted a fresh
     `in_flight` row → caller proceeds to run the handler.
   - `{ outcome: "replay", status, body }` — a row already exists with `status:"completed"` →
     caller short-circuits and returns the stored response.
   - `{ outcome: "in_flight" }` — row exists but still `in_flight` → caller returns `409
     conflict` ("a request with this Idempotency-Key is already being processed").
   - `{ outcome: "mismatch" }` — row exists with a different `requestHash` → `422 unprocessable`
     ("Idempotency-Key reused with a different request body").
   Implement by attempting the insert first; if `rowCount === 0`, `SELECT` the existing row and
   branch on its `status`/`requestHash`. **No `FOR UPDATE`, no `db.transaction()`** — the unique
   index + `ON CONFLICT DO NOTHING` is the whole concurrency primitive.
3. `completeIdempotencyKey(db, { apiKeyId, key, status, body })` — update the claimed row to
   `completed` with the captured status+body. Idempotent update (safe to call once on success).
4. `sweepExpired(db, olderThan)` helper (called opportunistically or via a future cron) deleting
   rows past the 24h TTL — bounded storage.
5. Tests (against a test DB or a mock honouring unique-constraint semantics): two concurrent
   claims of the same key → exactly one `claimed`, the other `in_flight`; after `complete`, a
   third claim → `replay` with the stored status+body; a different body under the same key →
   `mismatch`.

**Verify:** `pnpm -F @wherabouts.com/api test idempotency` green; grep confirms no
`db.transaction` / `for update` in the module.

**Commit:** `feat(api): idempotency claim/complete via ON CONFLICT upsert (no transactions)`

</task>

## Task 3 — idempotencyMiddleware in the write path

**Files:** `packages/api/src/routers/public-middleware.ts`

1. Export `idempotencyMiddleware`. It only acts when the request method is a write
   (`POST`/`PUT`/`DELETE`) **and** an `Idempotency-Key` header is present. For GETs or absent
   keys → pass straight through (writes without a key are still allowed; the server must not
   `400` on absence, matching the SDK contract).
2. Flow:
   - Compute `requestHash`; call `claimIdempotencyKey`.
   - `replay` → return the stored `{ status, body }` without invoking the handler. (Set the
     `X-Request-Id` from 11-01 fresh; the *body* is replayed.)
   - `in_flight` → throw `ORPCError("CONFLICT")`.
   - `mismatch` → throw `ORPCError("UNPROCESSABLE_CONTENT")`.
   - `claimed` → run the handler; on success call `completeIdempotencyKey` with the serialised
     result; on handler error, leave the row claimable again (delete the in_flight row or mark it
     failed) so a genuine retry can succeed — do **not** permanently poison the key on a 5xx.
3. Place it after `apiKeyAuth` (needs `apiKeyId`) and around the handler. Wire it into the public
   routers that have writes (zones create/update/delete, webhooks create/delete/reactivate,
   devices pushLocation, geocode batch submit). Read-only routers don't need it.

**Verify:** in-process test — submit `POST /api/v1/zones` with `Idempotency-Key: k1`; assert one
row created. Replay the identical request with `k1`; assert **no second row** and an identical
response body/status. A concurrent duplicate returns `409`.

**Commit:** `feat(api): enforce Idempotency-Key on writes (dedupe, replay, 409 in-flight)`

</task>
