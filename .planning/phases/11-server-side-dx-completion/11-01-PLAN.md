<objective>
Unify every public API error into the typed envelope the TS SDK already expects —
`{ error: { code, message, request_id, doc_url, fields? } }` — and stamp `X-Request-Id` on
every response (success and error). Widen the server error-code vocabulary to match the SDK's
`WheraboutsErrorCode` union and translate Zod validation issues into structured `fields[]`.

Purpose: The SDK ships `WheraboutsApiError.requestId/docUrl/fields` and a header-fallback test,
but the server never emits those keys — buyers get a degraded error experience and no
correlation id. This plan makes Success Criterion 1 TRUE and is the foundation the rate-limit
(11-02) and idempotency (11-03) error paths build on.

Output: Widened `api-response.ts`, a request-id helper, an upgraded response post-processor in
the Worker entry, and a round-trip test proving the published SDK parses a real error.
</objective>

## Task 1 — Widen the error envelope contract in `api-response.ts`

**Files:** `packages/api/src/api-response.ts`

1. Widen `ApiErrorCode` to the full SDK union: `bad_request | conflict | forbidden |
   internal_error | not_found | rate_limited | timeout | unauthorized | unprocessable`. This is
   the single server-side source of truth; it must stay `===` to the SDK's `WheraboutsErrorCode`
   in `packages/sdk/src/shared-types.ts`.
2. Extend `ApiErrorResponseBody.error` with optional `request_id?: string`, `doc_url?: string`,
   and `fields?: { path: string; message: string }[]` (snake_case on the wire — the SDK reads
   `request_id`/`doc_url`).
3. Add `errorDocUrl(code: ApiErrorCode): string` → `https://docs.wherabouts.com/errors/${code}`.
4. Extend `createApiErrorBody(code, message, opts?)` with optional `{ requestId, fields }`; it
   always sets `doc_url` via `errorDocUrl(code)`. Keep `jsonApiError` working (add an optional
   trailing opts arg threaded through).

**Verify:** `pnpm -F @wherabouts.com/api typecheck` passes; the union literally lists all 9
codes; `createApiErrorBody("not_found","x").error.doc_url` ends with `/errors/not_found`.

**Commit:** `feat(api): widen error envelope with request_id, doc_url, fields`

</task>

## Task 2 — Request-id helper

**Files:** `apps/server/src/request-id.ts` (create)

1. Export `resolveRequestId(req: Request): string` — return the inbound `X-Request-Id` if
   present and non-empty, else `cf-ray` if present, else `crypto.randomUUID()`.
2. No new dependency — `crypto.randomUUID()` is available in the Workers runtime.

**Verify:** unit test (can live in `index.test.ts`): given a request with `X-Request-Id: abc`
returns `abc`; given none returns a v4-shaped uuid.

**Commit:** `feat(server): add resolveRequestId helper`

</task>

## Task 3 — Stamp X-Request-Id + enrich error bodies in the response post-processor

**Files:** `apps/server/src/index.ts`

1. Compute `requestId = resolveRequestId(context.req.raw)` once at the top of the public
   `/api/v1/*` request handling and make it available to the post-processor.
2. **Every** response leaving the public router (success and error) must get
   `response.headers.set("X-Request-Id", requestId)`. Add it in the shared post-processing step,
   not per-route.
3. In the error branch (where `ORPC_TO_API_ERROR` is consulted):
   - Widen `ORPC_TO_API_ERROR` to map the new oRPC codes: `CONFLICT → {409, conflict}`,
     `FORBIDDEN → {403, forbidden}`, `UNPROCESSABLE_CONTENT`/`UNPROCESSABLE_ENTITY → {422,
     unprocessable}`, `TOO_MANY_REQUESTS → {429, rate_limited}`, keep existing
     `BAD_REQUEST/NOT_FOUND/UNAUTHORIZED/INTERNAL_SERVER_ERROR`.
   - Build the body with `createApiErrorBody(errorCode, message, { requestId, fields })` so it
     carries `request_id` + `doc_url` (and `fields` when present).
4. **fields extraction:** when the oRPC error is a `BAD_REQUEST` carrying Zod issues (oRPC
   exposes `data`/`issues` on validation errors), map each issue to `{ path:
   issue.path.join("."), message: issue.message }`. If no structured issues are available, omit
   `fields`.
5. Keep `cache-control: no-store` on error responses.

**Verify:** local `wrangler dev` (or the existing fetch-based handler test) —
`GET /api/v1/zones/999999` with a valid key returns `404` with body
`{ error: { code:"not_found", message, request_id, doc_url } }` and an `X-Request-Id` header
equal to the body's `request_id`. A bad input returns `400` with a non-empty `fields[]`.

**Commit:** `feat(server): stamp X-Request-Id and emit request_id/doc_url/fields on all errors`

</task>

## Task 4 — SDK round-trip test (server error -> WheraboutsApiError)

**Files:** `apps/server/src/index.test.ts` (extend or create)

1. Drive the Hono app in-process (the codebase already uses `localFetch` to route in-process,
   avoiding CF error 1042 — reuse that pattern). Issue a request that produces a `404` and a
   request that produces a `400` validation error.
2. Feed each raw `Response` through the SDK's error parsing by constructing a
   `createRequester({ apiKey, fetch: appFetch })` and asserting the rejected
   `WheraboutsApiError` has: `.status`, `.code` (from the union), `.requestId` (matches the
   `X-Request-Id` header), `.docUrl` ending `/errors/<code>`, and `.fields` populated for the
   400 case.
3. This is the proof that the server contract and the published client agree — guard against
   future drift.

**Verify:** `pnpm -F server test` (or the relevant workspace) green.

**Commit:** `test(server): assert SDK WheraboutsApiError round-trips the live error envelope`
