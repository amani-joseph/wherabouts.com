`RateLimit-Limit` / `RateLimit-Remaining` / `RateLimit-Reset` headers on every public response

(unit + a 429/Retry-After integration check against the SDK).

## Task 2 ‚Äî Fixed-window KV limiter module

**Files:** `packages/api/src/rate-limit.ts`, `packages/api/src/rate-limit.test.ts`

1. Implement `consume({ kv, apiKeyId, now, windowSeconds, max })` returning
   `{ limit, remaining, resetSeconds, retryAfter, allowed }`:
   - Window bucket key: `rl:{apiKeyId}:{floor(now/windowSeconds)}`.
   - Read current count from KV; if `>= max` ‚Üí `allowed:false`, `remaining:0`,
     `retryAfter = resetSeconds` (>=1).
   - Else increment (write count with `expirationTtl = windowSeconds` so buckets self-clean) and
     return `allowed:true`, `remaining = max - newCount`.
   - `resetSeconds = windowSeconds - (now % windowSeconds)`.
2. **Document the consistency caveat in a comment:** KV is eventually consistent across colos, so
   this is a coarse abuse guard, not a strict limiter (CONTEXT open question #2). Keep the module
   pure/injectable (pass `kv` + `now`) so it unit-tests without a real Worker.
3. Tests: under-limit returns decreasing `remaining`; at-limit returns `allowed:false` +
   `retryAfter >= 1`; a fresh window resets the count. Use a fake KV (`Map`-backed) honouring
   `expirationTtl` semantics loosely.

**Verify:** `pnpm -F @wherabouts.com/api test rate-limit` green.

**Commit:** `feat(api): fixed-window KV rate limiter (RateLimit-* + Retry-After math)`

</task>

## Task 3 ‚Äî rateLimitMiddleware in the public chain

**Files:** `packages/api/src/routers/public-middleware.ts`

1. Export `rateLimitMiddleware` (oRPC middleware via `baseBuilder.middleware`). It must run
   **after `apiKeyAuth`** (so `validatedApiKey.apiKeyId` is on context) and **before** the
   handler. If `context.env?.RATE_LIMIT` is absent (e.g. local dev without KV), no-op gracefully
   ‚Äî never hard-fail requests because the binding is missing.
2. Call `consume(...)` with `env.RATE_LIMIT`, `validatedApiKey.apiKeyId`, `Date.now()/1000`, and
   the env window/max. Stash the resulting `{ limit, remaining, resetSeconds, retryAfter }` on
   context (or a request-scoped holder) so the **server post-processor can set the headers on the
   final response** ‚Äî including on the handler's success response.
3. If `allowed === false`, throw `ORPCError("TOO_MANY_REQUESTS", { message: "Rate limit
   exceeded." })`. (11-01 already maps `TOO_MANY_REQUESTS ‚Üí 429/rate_limited`.) The thrown error
   path must still carry the `RateLimit-*` + `Retry-After` headers.
4. In `apps/server/src/index.ts` post-processor: read the stashed limiter result and set
   `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` on every public response; on the
   `429`, additionally set `Retry-After: <retryAfter>`.
5. Wire `rateLimitMiddleware` into every public router's chain
   (`.use(apiKeyAuth).use(rateLimitMiddleware).use(usageMiddleware(...))`). Prefer a small shared
   helper/composed builder so all routers pick it up uniformly.

**Verify:** in-process app test ‚Äî N allowed requests show `RateLimit-Remaining` counting down;
the (N+1)th returns `429`, body `code:"rate_limited"`, header `Retry-After` integer >= 1, and
`RateLimit-Remaining: 0`.

**Commit:** `feat(api): enforce per-key rate limit and emit RateLimit-*/Retry-After headers`

</task>
