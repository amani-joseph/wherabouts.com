  <name>Task 2: Add try-catch to fetchAuth in root route beforeLoad</name>
  <files>apps/web/src/routes/__root.tsx</files>
  <action>
Fix the root route's `beforeLoad` so that a thrown error from `fetchAuth()` does not kill navigation silently.

**Modify the `beforeLoad` function** in `apps/web/src/routes/__root.tsx`. Change from:

```typescript
beforeLoad: async (ctx) => {
  const token = await fetchAuth();
  if (token) {
    ctx.context.convexQueryClient.serverHttpClient?.setAuth(token);
  }
  return { isAuthenticated: !!token, token };
},
```

To:

```typescript
beforeLoad: async (ctx) => {
  let token: string | null = null;
  try {
    token = await fetchAuth();
  } catch {
    // fetchAuth may throw when getToken() lacks request context
    // (e.g., client-side SPA transitions). Default to unauthenticated.
  }
  if (token) {
    ctx.context.convexQueryClient.serverHttpClient?.setAuth(token);
  }
  return { isAuthenticated: !!token, token };
},
```

Key points:
- Declare `token` with `let` before the try block so it is accessible after
- Type it as `string | null` to match the existing interface
- Empty catch block is intentional — when `getToken()` throws (no server request context during SPA navigation), the user is simply treated as unauthenticated
- The `_protected.tsx` guard will then redirect them to sign-in, which is the correct behavior
- Do NOT add `console.log` or `console.error` in the catch block — the project convention is silent error handling for expected failures

**Do NOT change anything else in this file** — the imports, the `fetchAuth` server function definition, the `RouterAppContext` interface, the `head()` config, and the `RootDocument` component must all remain as-is.
