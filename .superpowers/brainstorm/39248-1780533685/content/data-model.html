Three decisions baked in:
1. Zones are per-project — scoped by project_id, same as API keys. No cross-project leakage.
2. Device state in Postgres, not KV — device_zone_state stores last-known zone membership as an integer array. Upsert on each location push. Avoids KV cold-start lag and keeps audit trail in one place.
3. Batch results in R2 — large result sets (thousands of rows) stored as JSON in Cloudflare R2, job record holds the key. Avoids Postgres row bloat and gives a clean download URL pattern.

Also fixes an existing issue: addresses.latitude/longitude are currently real (float32 — ~6 decimal places). All new spatial fields use double precision (float64). The addresses columns should be migrated too.
