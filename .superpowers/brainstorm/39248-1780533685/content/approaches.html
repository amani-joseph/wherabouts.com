Three approaches to building this out

All three deliver the same end state — they differ in sequence, risk profile, and how fast each capability ships.

A

Phased — geocoding first

Sequence: Forward geocoding → Batch jobs → Zones CRUD + PIP → Addresses-in-zone → Webhooks + device events

Ship value at each step. Each phase is independently useful. Webhooks (the hardest part) come last when the foundation is solid.

Pros

  • Lowest risk — each phase validates before the next
  • Forward geocoding ships fast (just a query change)
  • Webhooks scoped after real usage informs design

Cons

  • Geofencing lands later
  • Some rework if geocoding assumptions don't fit zones model
B

Polygon-first — shared spatial layer (Recommended)

Sequence: Zones data model (PostGIS polygons) → PIP + CRUD → Forward geocoding + context enrichment → Batch → Webhooks

Build the polygon primitives once. Forward geocoding, context enrichment, and addresses-in-zone all fall out naturally. No rework when adding new capabilities.

Pros

  • Single data model powers all three feature clusters
  • Context enrichment is nearly free once zones exist
  • Cleanest long-term architecture

Cons

  • Geofencing ships before forward geocoding
  • More upfront schema design
C

Parallel tracks — fastest calendar time

Sequence: Design full data model upfront → Track 1: geocoding + batch in parallel with Track 2: zones CRUD + PIP → Merge: webhooks + context enrichment

Two workstreams run simultaneously. Fastest wall-clock time to full feature set if you can run agents in parallel.

Pros

  • Everything ships sooner
  • Works well with GSD parallel execution

Cons

  • Integration risk at merge point
  • Requires upfront full-model design to avoid conflicts

Key edge case — webhooks

Entry/exit webhooks require devices to push location updates to Wherabouts. This adds a new endpoint: POST /api/v1/devices/ { id}/location. Each update checks all zones and fires webhooks on boundary crossings. The question is delivery: Cloudflare Queues (reliable async, ~1–3s delay) or Durable Objects (stateful, sub-second). Queue-based is simpler and fits the Workers architecture already in place.
