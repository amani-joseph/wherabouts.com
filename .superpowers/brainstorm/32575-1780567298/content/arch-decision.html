First decision: how does the dashboard talk to the new features?

This shapes everything else. The new geocoding/geofencing endpoints live in the public HTTP API (API-key auth, for external developers). The dashboard currently uses a separate typed oRPC router (session/cookie auth). They don't connect today.

Option A — New dashboard oRPC procedures

appRouter.zones.list() ...
Dashboard UI
→ orpcClient.zones.list()
→ appRouter (session auth)
→ same DB tables

Add session-authed procedures to the dashboard router that hit the same zones/webhooks/jobs tables. Mirrors how api-keys page already works.

Pros

  • End-to-end TypeScript types (no manual typing)
  • Uses logged-in session — no API key juggling
  • Consistent with existing dashboard pages
  • Can scope to user's projects cleanly

Cons

  • Duplicates handler logic (public HTTP + oRPC)
  • More backend code to build + keep in sync

Option B — UI calls public API with a key

fetch(/api/v1/zones, X-API-Key)
Dashboard UI
→ fetch(api.wherabouts.com)
→ publicHttpRouter (API-key auth)
→ same DB tables

The dashboard picks one of the user's API keys and calls the live public endpoints directly — same path external developers use.

Pros

  • Zero new backend code — reuses shipped endpoints
  • Dogfoods the real public API exactly
  • Fastest to build

Cons

  • UI must hold/select an API key in-browser
  • Manual response typing (or import SDK types)
  • Counts against the user's own usage/limits
  • CORS + key-exposure considerations
A

Dashboard oRPC procedures (Recommended)

Typed, session-authed, consistent with the existing dashboard. More backend work but the cleanest UX and safest.

B

Call public API with a key

Fastest, dogfoods the real API, but the dashboard has to manage an API key client-side and eats the user's usage.

C

Hybrid

oRPC for management (zones/webhooks CRUD, jobs), but a live "API Explorer / Try-it" panel that calls the real public API with a key so users see exactly what external devs get.
