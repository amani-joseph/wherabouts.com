❌ Still open — by priority

P1Client / mobile SDKs (iOS/Android/RN)ABSENT
Radar's core moat: drop-in background location + on-device geofencing. The defining gap for consumer/mobile geofencing. Multi-quarter; own milestone.
P3Server-side DX completionSDK sends, server doesn't emit
Verified live: error envelope is still bare { code,message} — no requestId / docUrl / fields; no rate-limit headers; idempotency not enforced. The SDK already sends these signals; the server must emit/enforce them to land the resilience story.
P3Python SDKABSENT
Mirror the now-proven namespaced TS surface. Server/data buyer can be won today without mobile SDKs.
P4Address validation / standardisationABSENT
Common buyer ask (Google/Smarty). Our G-NAF authority is a head-start we can do better than Mapbox/Google for AU — strongest data-moat play remaining.
P4Places / POI category searchABSENT
Search is address-only. "Find cafes near X" (Radar Places / Mapbox Search Box). Needs an AU POI dataset into the tiered-search infra.
P4Static Images APIABSENT
Server-rendered map PNG (markers/paths) for emails/reports/notifications — high utility with zero client SDK. Fits the self-hosted Protomaps stack.
P4Scoped / restricted API tokensABSENT
URL/scope/IP restrictions + per-key usage breakdowns (Mapbox Tokens). DX maturity; not present in api-key-auth.

The take

Routing was the biggest competitive hole and it's now closed — Wherabouts has reached geocoding + geofencing + routing parity on the AU-authoritative-data moat. The remaining gaps cluster into two cheap, high-leverage moves and one big one:
  • P0 — publish the SDK ✓ DONE (@wherabouts/sdk@0.2.0 live on npm; set a real license — still UNLICENSED).
  • Now top priority — finish server DX (error envelope, rate-limit headers, idempotency) — the SDK already speaks the protocol; the server must emit it.
  • P4 — address validation — the play only our G-NAF authority can win.
The defining strategic gap stays client/mobile SDKs (Radar's moat) — still multi-quarter.
